# 아키텍처 문서

## 목적
이 저장소는 하네스 엔지니어링을 위한 정본 문맥을 담는다. 여기에는 제품 의도, 설계 결정, 실행 계획, 운영 지침이 포함된다.

## 저장소 지도
- `AGENTS.md`: 기여자와 에이전트를 위한 운영 규칙
- `docs/DESIGN.md`: 의사결정 원칙과 설계 기준
- `docs/FRONTEND.md`: 인터페이스와 UX 규칙
- `docs/GLOSSARY.md`: 하네스 업무 흐름에서 쓰는 공통 용어
- `docs/CHECKLISTS.md`: 인테이크부터 완료까지의 단계별 확인 기준
- `docs/PLANS.md`: 실행 계획 형식과 작업 방식
- `docs/PRODUCT_SENSE.md`: 제품 관점과 우선순위 판단 기준
- `docs/QUALITY_SCORE.md`: 검토 기준표
- `docs/RELIABILITY.md`: 가용성과 운영 기준
- `docs/SECURITY.md`: 보안 기준선
- `docs/design-docs/`: 오래 유지되는 설계 메모와 원칙
- `docs/product-specs/`: 사용자 관점 요구사항과 제품 명세
- `docs/exec-plans/`: 진행 중 및 완료된 실행 계획
- `docs/planning/`: 기획팀 운영 공간, 사람별 Jira 큐, 공용 정책
- `docs/generated/`: 기계가 생성한 산출물
- `docs/references/`: 반복 사용되는 도구와 관행에 대한 간결한 참조 파일
- `docs/skills/`: 반복 가능한 작업 절차와 검증 기준을 담는 문서

## 정보 흐름
1. 문제를 제품 명세로 기록한다.
2. 합의된 접근법을 설계 문서로 옮긴다.
3. 작업을 실행 계획으로 쪼갠다.
4. 공통 용어는 용어집으로 맞추고, 단계별 확인은 체크리스트로 맞춘다.
5. 작업이 진행되는 동안 구현 세부사항과 결정을 추적한다.
6. 안정적이고 반복되는 사실은 생성 문서나 참조 문서로 옮기고, 반복되는 작업 절차는 스킬 문서로 분리한다.
7. 결과, 교훈, 남은 부채를 기록하며 마무리한다.

## 경계
- 제품 명세는 무엇이 필요하고 왜 필요한지를 설명한다.
- 설계 문서는 그것을 어떻게 풀려는지 설명한다.
- 실행 계획은 누가, 언제, 무엇을 하고, 어떻게 검증할지를 설명한다.
- 생성 문서는 계산된 사실을 담으며 손으로 편집하지 않는다.
- 참조 파일은 도구와 미래 독자를 위한 간결하고 지속적인 문맥을 담는다.
- 스킬 문서는 반복 가능한 작업 절차와 검증 방법을 담는다.

## 갱신 규칙
- 어떤 섹션에 새 문서를 추가하면 해당 디렉터리의 인덱스도 함께 갱신한다.
- 중복보다 링크를 우선한다.
- 의견성 문서에는 생성된 사실을 넣지 않는다.
- 아키텍처가 바뀌면, 이유와 마이그레이션 경로를 관련 설계 문서나 계획 문서에 기록한다.
