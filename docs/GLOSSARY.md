# 용어집

## 목적
하네스 엔지니어링 작업에서 쓰는 공통 용어를 고정해 문서와 대화를 일관되게 유지한다.

## 사용 원칙
- 같은 개념은 같은 이름으로 쓴다.
- 새 약어를 만들기 전에 기존 용어로 설명할 수 있는지 먼저 본다.
- 애매한 용어는 이 문서에 먼저 정의한다.
- 제품 명세, 설계 문서, 실행 계획에서 같은 용어를 같은 의미로 쓴다.

## 핵심 용어
| 용어 | 의미 | 주로 쓰는 곳 |
| --- | --- | --- |
| 인테이크 | 요청을 처음 받아 문제를 정리하는 단계 | 초기 정리, 계획 수립 |
| 정본 | 어떤 사실에 대해 가장 권위 있는 문서나 파일 | 아키텍처, 참조, 생성 문서 |
| 제품 명세 | 무엇을 왜 바꿔야 하는지 설명한 문서 | `docs/product-specs/` |
| 설계 문서 | 어떻게 풀지와 그 트레이드오프를 설명한 문서 | `docs/design-docs/` |
| 실행 계획 | 누가, 언제, 무엇을, 어떻게 검증하며 할지 적은 문서 | `docs/exec-plans/active/` |
| 활성 계획 | 현재 진행 중인 실행 계획 | `docs/exec-plans/active/` |
| 완료 계획 | 종료되었거나 보관된 실행 계획 | `docs/exec-plans/completed/` |
| 체크리스트 | 단계별로 빠짐없이 확인해야 하는 항목 묶음 | `docs/CHECKLISTS.md` |
| 가정 | 아직 검증되지 않았지만 전제로 둔 내용 | 명세, 설계, 계획 |
| 열린 질문 | 답이 아직 없는 사항 | 명세, 설계, 계획 |
| 위험 | 실패하거나 지연될 수 있는 조건 | 설계, 계획, 신뢰성 |
| 롤백 경로 | 변경을 되돌리거나 끄는 방법 | 계획, 신뢰성 |
| 검증 | 의도한 결과가 실제로 나왔는지 확인하는 근거 | 계획, 완료 보고 |
| 영향 범위 | 실패 시 어디까지 영향을 주는지 | 신뢰성, 보안 |
| 오너 | 결과에 책임지는 사람 | 계획, 부채 추적 |
| 기술 부채 | 지금은 미루지만 나중에 반드시 정리할 항목 | `docs/exec-plans/tech-debt-tracker.md` |
| 생성 문서 | 도구가 만든 산출물로 손으로 편집하지 않는 문서 | `docs/generated/` |
| 참조 파일 | 반복해서 쓰는 기준을 짧게 정리한 파일 | `docs/references/` |

## 흐름 용어
1. 인테이크에서 요청과 문제를 정리한다.
2. 제품 명세에서 가치와 범위를 정한다.
3. 설계 문서에서 접근법과 트레이드오프를 고른다.
4. 실행 계획에서 작업 단위와 검증 방법을 쪼갠다.
5. 진행 중에는 결정, 위험, 차단 요소를 계속 갱신한다.
6. 완료 후에는 결과, 검증, 후속 작업을 기록한다.
