# SPAG-6823 BOS 보관정보 실온 정규화

이 문서는 SPAG-6823의 남은 구현, 배포, 검증을 추적한다.

## 목적
- ERP에서 내려오는 보관방법 값의 `상온/실온` 혼재를 `실온`으로 정규화한다.
- BOS에서 `상온` 선택지를 더 이상 운영값으로 사용하지 않도록 정리한다.
- 정기 배포에 포함하고, 배포 후 실제 저장 결과를 확인한다.

## 배경
- 원본 이슈는 `SPAG-6791`, 개발 이슈는 `SPAG-6823`, 연계 상위 이슈는 `SPMO-7285`이다.
- 2026-04-07 댓글 기준으로 요청 1은 정기 배포 포함 요청이 있었고, 요청 2는 2026-04-08에 미사용 처리 완료가 확인됐다.
- 따라서 이 계획은 요청 1의 반영과 최종 검증을 중심으로 관리한다.

## 범위
- ERP 품목정보가 BOS로 유입될 때 보관방법 값이 `상온` 또는 `실온`이면 최종 저장값을 `실온`으로 치환한다.
- BOS 코드 관리에서 `ERP_STORAGE_TYPE.ORDINARY`를 비활성화하거나, 최소한 운영 경로에서 선택 불가 상태로 둔다.
- 정기 배포 반영 여부와 배포 후 결과를 검증한다.

## 비범위
- 기존 저장 데이터 전체의 일괄 백필
- `냉장`, `냉동`, `기타` 값의 의미 변경
- 이 이슈와 무관한 다른 코드값 정리

## 가정
- 신규 저장 또는 수정 경로에서 `실온`으로 수렴하면 운영 요구를 충족한다.
- `ERP_STORAGE_TYPE.ORDINARY`를 더 이상 노출하지 않아도 다른 기능이 깨지지 않는다.
- 기존 데이터의 역사적 값 보존이 필요하다면 별도 후속 작업으로 분리한다.

## 열린 질문
- 기존에 저장된 `상온` 데이터도 별도 백필이 필요한가.
- `ORDINARY`를 완전 삭제할지, 비활성화 상태로 보존할지.

## 위험
- 다른 저장 경로가 남아 있으면 `상온` 값이 재유입될 수 있다.
- 운영 코드값을 숨기기만 하고 저장 로직을 바꾸지 않으면 데이터가 다시 어긋날 수 있다.
- 정기 배포에 포함되지 않으면 요청 1과 요청 2의 상태가 분리되어 혼선을 만들 수 있다.

## 롤백 경로
- 저장 로직 변경은 직전 배포 버전으로 되돌릴 수 있어야 한다.
- `ORDINARY` 비활성화가 문제를 만들면, 재노출 전 영향 범위를 먼저 확인한다.
- 롤백 후에는 동일 경로로 다시 `상온`이 저장되는지 재검증한다.

## 검증
- ERP 연동값이 `상온`인 샘플을 저장했을 때 BOS 최종 저장값이 `실온`인지 확인한다.
- ERP 연동값이 `실온`인 샘플도 동일하게 `실온`으로 저장되는지 확인한다.
- BOS 코드 관리 화면에서 `상온` 옵션이 선택 가능 상태로 남지 않았는지 확인한다.
- `냉장`, `냉동`, `기타` 저장값은 기존과 동일하게 동작하는지 확인한다.

## 완료 기준
- 요청 1이 배포된 상태로 운영 반영된다.
- 요청 2의 미사용 처리 상태가 유지된다.
- 신규 저장 경로 어디에서도 `상온`이 활성 운영값으로 남지 않는다.
- 배포 후 검증 결과와 남은 후속 항목이 기록된다.
