# 실행 계획 작성

## 목적
문제, 명세, 설계 내용을 실제 실행 계획으로 빠르게 전환하는 절차를 정의한다.

## 사용 시점
- 제품 명세나 설계 문서가 이미 있고 작업 순서를 정해야 할 때
- 위험, 의존성, 검증, 롤백이 필요한 변경을 시작할 때
- 여러 사람이 같은 계획을 보고 작업 상태를 맞춰야 할 때

## 입력
- 관련 제품 명세
- 관련 설계 문서
- Jira 이슈 또는 작업 요청
- 필요한 경우 [용어집](../GLOSSARY.md)과 [체크리스트](../CHECKLISTS.md)

## 절차
1. 이 작업이 실행 계획이 필요한지 먼저 판단한다.
2. 문제, 목표, 범위를 한 문장씩 정리한다.
3. 비목표와 가정을 적는다.
4. 실제 실행 순서대로 마일스톤을 나눈다.
5. 각 마일스톤에 검증 방법을 붙인다.
6. 위험, 롤백 경로, 남은 질문을 적는다.
7. 관련 문서 링크를 연결한다.
8. 계획을 읽는 사람이 다음 행동을 바로 알 수 있게 정리한다.

## 작성 기준
- 마일스톤은 실제로 끝낼 수 있는 단위여야 한다.
- 검증은 관찰 가능한 결과로 써야 한다.
- 위험은 실패 형태까지 드러나야 한다.
- 롤백은 가능한 경우 되돌리는 조건도 적어야 한다.
- 계획 문서는 작업 지시서가 아니라 실행 가능한 합의문이어야 한다.

## 검증
- 목표, 범위, 비목표가 서로 충돌하지 않는지 본다.
- 마일스톤이 구현 순서로 읽히는지 본다.
- 각 마일스톤에 검증이 붙어 있는지 본다.
- 위험과 롤백이 빠지지 않았는지 본다.
- 관련 문서 링크가 모두 연결되는지 본다.

## 예외와 주의점
- 작업이 너무 작아서 계획이 과하면, 관련 문서에 링크만 남긴다.
- 실제 이슈 고유 내용은 이 스킬이 아니라 실행 계획 문서에 적는다.
- 새 계획을 만들기 전에 기존 활성 계획이 있는지 확인한다.
