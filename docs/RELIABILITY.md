# 신뢰성

## 목적
실패하거나, 성능이 저하되거나, 나중에 호출을 유발할 수 있는 작업에는 이 가이드를 사용한다.

## 신뢰성 원칙
- 핵심 경로를 먼저 정의한다.
- 반복이 가능한 곳에서는 멱등성을 설계한다.
- 타임아웃과 합리적인 재시도로 지연 시간을 제한한다.
- 조용히 실패하지 말고 눈에 보이게 실패한다.
- 가능한 경우 전체 실패보다 점진적 저하를 선호한다.
- 운영에서 의존하기 전에 행동을 계측한다.
- 복구 절차는 단순하고 문서화돼 있어야 한다.

## 필요한 운영 관점
- 실패 범위는 무엇인가?
- 사용자에게 보이는 영향은 무엇인가?
- 문제는 어떻게 감지하는가?
- 어떻게 복구하는가?
- 영향 범위는 얼마나 넓은가?
- 부하가 걸리거나 일부 장애가 났을 때는 어떻게 되는가?

## 기본 체크리스트
- 핵심 경로에 대한 지표가 있다
- 로그가 실행 가능하고 민감한 데이터를 누설하지 않는다
- 알림이 의미 있는 사용자 영향과 연결돼 있다
- 롤백 또는 비활성화 경로가 문서화돼 있다
- 속도 제한, 백프레셔, 큐 한도가 고려돼 있다
- 타임아웃과 재시도가 명시돼 있다
- 용량 가정이 문서화돼 있다

## 사고 대응 준비
변경이 가용성이나 처리량에 영향을 줄 수 있다면 계획에 다음이 포함돼야 한다:
- 감지
- 분류
- 완화
- 롤백
- 후속 조치 항목
