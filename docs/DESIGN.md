# 설계 기준

## 목적
이 문서는 시스템, 워크플로, 문서 집합을 설계할 때 사용하는 기준을 정의한다.

## 원칙
- 가장 가치가 큰 사용자 문제부터 해결한다.
- 영리함보다 명시성을 우선한다.
- 실패 모드는 보이게 만들고 복구 가능해야 한다.
- 단순한 경계와 작은 인터페이스를 선호한다.
- 오래 가는 규칙과 임시 계획을 분리한다.
- 모든 설계 선택에는 이름 붙은 트레이드오프가 있어야 한다.
- 구현 속도뿐 아니라 검토 가능성을 위해 설계한다.
- 미래의 유지보수자가 1분 안에 설명하지 못한다면, 너무 복잡한 것이다.

## 설계 문서 필수 섹션
- 문제
- 맥락
- 목표
- 비목표
- 검토한 대안
- 권장안
- 위험과 완화책
- 롤아웃 또는 마이그레이션
- 열린 질문
- 참고 자료

## 설계 문서 품질
좋은 설계 문서는 다음 질문에 답해야 한다:
- 사용자나 운영자에게 무엇이 달라지는가?
- 왜 지금 해야 하는가?
- 하지 않으면 무엇이 깨지는가?
- 무엇이 잘못될 수 있는가?
- 설계가 잘 작동했는지 어떻게 알 것인가?
