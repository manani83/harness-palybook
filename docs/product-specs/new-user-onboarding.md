# 신규 사용자 온보딩

## 문제
새 기여자는 저장소를 어떻게 탐색하는지, 정본 문서를 어디서 찾는지, 어떤 문서에 무엇을 써야 하는지, 그리고 암묵지에 의존하지 않고 어떻게 일을 시작하거나 끝내는지 알아야 한다.

## 목표
- 저장소 전체를 한 번에 이해할 수 있게 한다.
- 결정, 계획, 생성 데이터를 어디에 두는지 보여준다.
- 처음으로 유용한 기여를 하기까지의 시간을 줄인다.
- 품질, 신뢰성, 보안을 어떻게 평가할지 분명하게 만든다.

## 비목표
- 완전한 교육 시스템을 만든다.
- 다른 곳에 문서화되지 않은 회사 전용 정책을 여기에 인코딩한다.
- 이 저장소 밖의 팀 온보딩을 대체한다.

## 주요 사용자
- 사전 맥락이 없는 새 엔지니어, 에이전트, 또는 리뷰어

## 원하는 흐름
1. `AGENTS.md`를 연다.
2. `ARCHITECTURE.md`를 읽는다.
3. `docs/GLOSSARY.md`로 용어를 맞춘다.
4. `docs/CHECKLISTS.md`로 단계별 확인 기준을 본다.
5. 관련 문서 유형을 찾는다.
6. 맞는 파일을 새로 만들거나 갱신한다.
7. 필요한 체크리스트로 변경 사항을 검증한다.
8. 결과와 후속 항목을 기록하며 마무리한다.

## 성공 지표
- 첫 올바른 편집까지 걸리는 시간이 짧다.
- 독자가 새 아이디어를 어디에 둬야 하는지 식별할 수 있다.
- 계획과 결정이 문서 전반에서 일관된다.
- 리뷰어가 빠진 맥락을 따로 물어보지 않아도 품질을 판단할 수 있다.

## 엣지 케이스
- 새 주제에 분명한 자리이 없다.
- 하나의 계획이 제품, 설계, 운영을 모두 아우른다.
- 생성된 사실을 손으로 쓴 결정으로 착각한다.

## 승인 기준
- 저장소가 각 정보 유형의 위치를 명확히 설명한다.
- 새로 들어온 사람도 문제에서 계획, 실행으로 이동할 수 있다.
- 문서 집합이 품질, 신뢰성, 보안 점검을 명시적으로 만든다.
